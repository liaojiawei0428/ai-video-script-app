## § 1. 五段式汇报模板 (适用所有场景, 跨项目通用)

**任何任务完成, 必跑这五段, 顺序固定, 不能跳**:

### 段 1: 这次干了啥 (大白话, 2-4 句话)

**模板**:
> 这次我 [做了什么事, 用日常语言]. 简单说就是 [用比喻/类比讲清功能, 像跟非技术朋友解释].

**示例**:
> 这次我修了一个用户改方案时点不动的按钮问题. 简单说就是, 之前服务器收到"用户要换个方案"的请求时, 系统会偷偷拒绝, 还报一个看不懂的"网络错误"; 现在我让服务器在这种情况下直接告诉用户"现在太忙, 请稍后再试", 用户就知道是等一下就行, 不是 应用坏了.

### 段 2: 为什么干这个 (1-2 句话, 说明重要性)

**模板**:
> 这个事重要因为 [业务影响 / 用户痛点 / 风险]. 之前 [不做的后果].

**示例**:
> 这个事重要因为, 用户改不了方案就没法继续生图 / 生视频, 整个产品核心流程被堵死. 之前这个问题已经被一个真实用户撞到, 如果不修, 下一个用户大概率还会撞.

### 段 3: 怎么知道干好了 (可观察的验证, 表格或列表)

**模板**:
```
用户能看到的现象:
- [生产页面 / 接口 / 数据 / 行为 变化 1]
- [生产页面 / 接口 / 数据 / 行为 变化 2]
```

**示例**:
```
用户能看到的现象:
- 生产部署后, 用户改方案时, 页面会显示"系统繁忙, 请稍候"而不是"网络错误"
- 后台看得到所有"系统繁忙"的拒绝事件, 用于后续分析活跃任务数
- 如果没活跃任务, 用户改方案立即成功, 跟以前一样
```

**不要**:
- ❌ "跑了 21 维自动检查全过" (用户不知道这 21 维是啥, 也不关心)
- ❌ "文本搜索命中 X 个文件" (技术细节)
- ❌ "编译检查 0 错" (用户不知道编译是啥)

### 段 4: 改了哪些东西 (文件路径列表, 一句话功能描述, 不贴代码)

**模板**:
| 文件 | 功能描述 (大白话) |
|---|---|
| `路径1` | 这个文件是 [负责什么业务], 改了 [这个具体功能/逻辑] |
| `路径2` | 这个工具是 [给其他程序用的辅助], 加了 [这个新检查/能力] |

**示例**:
| 文件 | 功能描述 |
|---|---|
| `apps/server/src/services/imageAgentService.ts` | 视频/图片 智能体的大脑, 决定下一步做什么. 加了"检查现在能不能接新任务"的逻辑 |
| `apps/server/src/utils/errors.ts` | 错误处理工具, 统一报错格式. 加了一种"系统繁忙"的错误类型 |
| `apps/web/src/components/AgentChatPanel.tsx` | 用户跟 AI 对话的界面. 改错误提示文案, 让用户看懂"现在太忙" |
| `apps/mobile/BUGS.md` | 记录所有踩过的坑. 加了 BUG-081 条目, 防止下次再撞 |

**不要**:
- ❌ 函数名 / 变量名 / 行号 / 改了哪一行
- ❌ 代码片段 (用户说过"写的好不好都是 AI 自己改")
- ❌ 详细的正则表达式 / 结构化查询语句

### 段 5: 下一步 (1-3 句话, 留待办)

**模板**:
> 接下来 [可能的下一步 / 我建议的事 / 等用户决定]. [如果用户没说, 我会 [默认行为]].

**示例**:
> 接下来还有 3 件事可选:
> 1. 修另一个问题 (BUG-082, 用户撞视频/图片会话崩)
> 2. 加新功能 (跟你定方向)
> 3. 我自己继续修类似的小问题
>
> 你拍, 我跟着干.
