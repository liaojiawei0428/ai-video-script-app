## 版本号格式

```
X.Y.Z
│ │ └─ 1类更新（小版本）：仅UI逻辑改动，不变动功能逻辑代码
│ └─── 2类更新（中版本）：UI改动 + 小功能微调，不改动核心功能
└───── 3类更新（大版本）：UI改动 + 核心功能改动
```

### 三类更新定义

| 类型 | 位置 | 说明 | 示例 |
|------|------|------|------|
| **1类（小版本）** | 末位 Z | UI样式调整、布局修复、图标替换、文案修改。**不改动任何功能逻辑和核心功能代码**，仅改动前端UI代码 | 1.0.1 → 1.0.2 |
| **2类（中版本）** | 中位 Y | 1类内容 + 小功能微调（如新增设置项、优化交互流程）。改动前端代码 + 小功能代码 | 1.0.2 → 1.1.0 |
| **3类（大版本）** | 首位 X | 1类+2类内容 + 核心功能新增或重大改动（如新增AI模型、架构变更）。基本涵盖全方位代码改动 | 1.1.0 → 2.0.0 |

### 进位规则

- 执行1类更新时：末位 Z + 1
- 执行2类更新时：中位 Y + 1，末位 Z 归零
- 执行3类更新时：首位 X + 1，中位 Y 和末位 Z 归零

## 版本号声明位置

**唯一版本号来源**：`apps/mobile/src/config/version.ts`

```typescript
export const APP_VERSION = 'X.Y.Z';
export const APP_NAME = 'Deep剧本';
export const APP_DISPLAY_NAME = `${APP_NAME} v${APP_VERSION}`;
```

所有需要版本号的地方（更新检查、关于页面、下载页、APK文件名）统一使用此导出。

---

## 发布前判定

每次发布前，按以下流程确定版本号：

```
1. 列出本次所有变更内容
2. 判定每项变更属于哪一类（1类/2类/3类）
3. 取最高类别作为本次更新类别
4. 按进位规则计算新版本号
5. 更新 version.ts 中的 APP_VERSION
```

## v2.0.0 类别判定详细说明

按 [版本管理规范] 第一节三类更新定义, v2.0.0 属于**3类(大版本)** — 首位升位, 理由如下:

| 变更项 | 类别 | 说明 |
|------|------|------|
| 角色一致性三阶段流程 | **3类** | 核心功能新增: AI 文字描述生成 + 用户确认 + 多角度变体图, 改变整体剧本生成流程 |
| 分集大纲确认 | **3类** | 核心流程变更: 插入新步骤(分集大纲生成→用户确认), 涉及 novel 状态机扩展 |
| 章节事件图谱 | **3类** | 新增 plotGraph 数据结构 + AI 生成, 注入所有后续 prompt, 改动 chunkService/scriptService 核心 |
| 5 画风预设系统 | **2类** | 新增配置项 + UI 组件, 不改动核心流程 |
| 多角度变体图 (3张/角色) | **3类** | 核心功能: 改 characterService + 新增 imageProvider 适配层 + 按张扣费机制 |
| PDF/Word 导出 | **2类** | 新增导出服务, 不改动核心生成流程 |
| 书架搜索/筛选 | **1类** | UI 交互优化, 不改动核心功能 |
| 积分订单系统 | **2类** | 新增订单数据模型 + 支付回调, 改动 billing 边缘 |
| Web 端(Vite+React18+TS) | **3类** | **架构变更**: 全新 monorepo 形态 + 新技术栈 + 无限画布 + AI 助手侧栏 |
| 共享 users 表 | **2类** | 跨端账户统一, 改动鉴权边缘 |
| 增量 DB 迁移 | **1类** | 兼容性 ALTER TABLE, 不破坏 v1.2.0 数据 |
| **最高类别** | **3类** | 核心功能 + 架构变更双重 3 类, 取最高 |

→ **v2.0.0 = 3类更新 = 首位 X+1, 中位 Y 和末位 Z 归零**
