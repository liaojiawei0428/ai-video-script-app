四、瓶颈分析

🔴 主要瓶颈：DeepSeek API 连接稳定性

90 槽位时出现 351 次 socket hang up / ECONNRESET。这不是服务器或代码问题，而是 DeepSeek API 在极高并发下的连接策略。3 个 Key 共享同一源 IP，API 端可能按 IP 维度限流。

五、优化建议

推荐配置：AI_MAX_CONCURRENT=10（30 总槽位）

配置值理由
AI_MAX_CONCURRENT10每 Key 10 槽位，总共 30 槽位。实测此为稳定值
CHUNK_CONCURRENT10与 AI 槽位对齐，单本全速跑完
分块大小400K适配 1M tokens 窗口，预留输出余量
max_tokens32768满足剧本输出需求，不浪费配额
推荐理由：30 总槽位是经过 4 轮迭代验证出的稳定值。
• P1（20本小小说）：30 槽位 → 20 块全并发，秒级完成 ✅
• P2（21混合）：30 槽位 → 126 块分 4-5 轮处理，~8 分钟完成 ✅
• P3（20本大雪中）：30 槽位 → 300 块分 10 轮处理，~20 分钟完成 ✅
30 槽位下 socket 错误概率极低，是最佳平衡点。

后续改进方向

  1. chunkService 超时重试优化：当前 axios timeout=300s 太长，建议降到 120s。3 次指数退避重试足够。
  2. mergeSummaries 容错：某本小说所有块失败时，mergeSummaries 会卡住。应增加空摘要兜底逻辑。
  3. HTTP 连接池限制：axios 默认无全局连接池限制。建议增加 maxSockets 限制匹配 AI 槽位数。
  4. API Key 层面隔离：3 个 Key 从同一 IP 发出，建议后续使用不同源 IP 的代理，绕过 IP 级限流。
